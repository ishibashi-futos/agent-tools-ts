## Goal

`specs/features/feat_tool_name.template.md` に準拠した、機能仕様ドラフトを 1 つ作成せよ。
出力先は `specs/features/feat_<tool_name>.md` とする。

対象機能:
- `tool_name`: `<TOOL_NAME>`
- 優先度: `<PRIORITY>`

## Context

- このリポジトリは AI エージェント向けツール群を提供する。
- Windows/MacOSでの利用を想定するため、パス区切り・エスケープについて機能的・descriptionなどでエージェント向けの考慮をする。
- 仕様は日本語で作成する。
- 初期版は「シンプルで唯一の正解」を優先し、過剰な後方互換・複雑な例外処理は避ける。
- 目的は、実装とテストに直結するドラフトを短時間で作ること。

## Source

必ず次の情報源を参照すること。

1. テンプレート:
   - `specs/features/feat_tool_name.template.md`
2. バックログ:
   - `specs/backlog.md`
3. 既存仕様（存在する場合）:
   - `specs/features/feat_*.md`
4. 実装コード（存在する場合）:
   - `src/` 配下の関連ツール
5. テストコード（存在する場合）:
   - `test/` 配下の関連テスト

不足情報があれば、推測ではなく `Open Questions` に明記すること。

## Expectations

次の要件をすべて満たして出力せよ。

1. 章構成
- テンプレートの章番号・順序（1〜11）を維持する。
- 章を削らない。不要な場合は「初期版では非適用」と記載する。

2. 内容品質
- MUST/SHOULD/WON'T を混在させない。
- Input/Output/Error は TypeScript 型または同等の構造で明示する。
- 受け入れ基準（DoD）は pass/fail 判定可能な文で書く。
- Open Questions は「未決事項」と「決定方法（どう決めるか）」をセットで書く。

3. 実装接続性
- `8. 実装分割` では、想定ファイルパスと責務を具体化する。
- 既存実装がある場合、責務重複や衝突を避ける方針を明記する。
- 将来拡張（`9`）は、他ツールへの接続点を 2 件以上示す。

4. 出力ルール
- 出力は日本語。
- 冗長な背景説明は避け、実装判断に必要な情報を優先する。
- 最後に「未確定事項サマリ」を 3 行以内で付ける。

5. 完了条件
- `specs/features/feat_<tool_name>.md` を新規作成または更新する。
- 変更内容の要約を 5 行以内で提示する。
